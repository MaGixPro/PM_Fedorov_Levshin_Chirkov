# Описание практики и задания

## Общие правила выполнения
- Работа выполняется в группе из **3 человек**.
- Для **каждой фазы** обязательно делается **декомпозиция**: разбиваем работу на отдельные задачи (и так для любой активности в рамках практики).
- Задачи распределяются между участниками, а распределение фиксируется:
  - либо в **матрице RACI** (одна общая матрица, её обновляем после каждого этапа),
  - либо в **таск-трекере**, где назначение исполнителей должно быть явно видно.
- Для каждой задачи обязательно указываются:
  - **период выполнения** (даты),
  - **примерное чистое время** (сколько реально потратили часов).

> Эти требования важны: часть следующих фаз опирается на вашу статистику задач и времени.

## Организационные замечания
- Чтобы проще вести статистику и не держать RACI “вручную”, можно использовать **любой таск-трекер**. Важно: проект должен быть публичным (или нужно добавить преподавателя) и ссылка должна быть в отчёте.
- Все артефакты (таблицы, документы, код, отчёты) удобно хранить в репозитории или облаке. Ссылку на хранилище добавляем в итоговые материалы.
- В финальном отчёте по практике должно быть всё по фазам: как выполнялось, какие промежуточные результаты получались, какие решения принимались, а также:
  - актуальная **RACI**,
  - логика распределения задач.
- В список задач включаем не только “кодинг”, но и любую деятельность: обсуждения, митинги, анализ теории, чтение материалов и т.п.
- В заданиях будет немного разработки, поэтому в команде должен быть человек, который уверенно поможет с кодом; обязанности стоит распределять разумно.
- Концепция проекта должна быть **компактной**: примерно **10–15 use case** (не меньше 10 и не больше 15). Можно формулировать как MVP с базовой функциональностью в этих рамках.
- Там, где требуется программирование, обычно есть доп. материалы (библиотеки, ссылки, примеры).
- Использование LLM допускается **в рамках требований задания** (как правило — генерация идей, помощь в аналитике и кодинге при понимании результата).

# Фазы практики

## Фаза 1 — Выбор концепции проекта
Нужно:
- придумать **минимум 4 концепции** программного продукта;
- задать **минимум 4 критерия** выбора (критерии могут описывать продукт или удобство/риски работы над ним в рамках дисциплины);
- провести **экспертную оценку**: каждый участник ставит трёхточечные значения (пессимистично / ожидаемо / оптимистично) для:
  - важности критериев,
  - оценок альтернатив по критериям;
- выбрать лучшую концепцию методами:
  - **нечёткий коллективный TOPSIS** (на трёхточечных оценках),
  - **чёткий коллективный МАИ (AHP)** (в качестве чёткой оценки использовать мат. ожидание трёхточечной);
- сравнить результаты двух методов и сделать вывод.

LLM разрешено использовать для генерации концепций и критериев (важно сразу учитывать, что дальше проект нужно будет оценивать и описывать use case).

---

## Фаза 2 (Мета) — Распределение работы
> Фазы 2 и 3 связаны: сначала продумываем структуру работ по фазе 3.
Требуется:
- разложить работы фазы 3 на задачи;
- оценить трудоёмкость задач экспертно (подойдёт любая практика оценивания);
- собрать индивидуальные предпочтения задач от каждого участника;
- сформулировать задачу распределения (целевая функция + ограничения) и обосновать постановку;
- решить задачу любым инструментом (OpenSolver / NEOS / Pyomo и т.д.).

LLM можно использовать как помощь в кодинге и в формулировках модели (но нужно понимать, что именно решается).

---

## Фаза 3 — Оценка проекта
Нужно для выбранного проекта:
- составить **не менее 10 use case**;
- выполнить оценку в **COSMIC Function Points**;
- выполнить оценку в **Use Case Points** и прикинуть трудозатраты;
- провести коллективную экспертную оценку:
  - оценить трудозатраты для каждого use case (или декомпозированных частей: user stories / slices),
  - агрегировать оценки до масштаба всего проекта;
- сравнить разные оценки и сделать вывод:
  - размер команды,
  - ориентировочные сроки,
  - какие функции можно убрать при необходимости (маловажные, но “тяжёлые”).

LLM можно использовать для генерации use case, но важно следить за альтернативными и негативными сценариями.

## Фаза 4 (Мета) — Прогнозирование
> Фазы 4 и 5 связаны: сначала готовим задачи для фазы 5.
Нужно:
- разложить фазу 5 на задачи;
- дать трёхточечные оценки длительности задач;
- спрогнозировать сроки методом Монте-Карло двумя способами:
  1) смоделировать выполнение по статистике “сколько задач закрываем за неделю” (дискретное распределение), оценить срок завершения фазы 5 с 80% уверенностью;
  2) смоделировать трудозатраты (человеко-часы в неделю) как дискретное распределение + длительности задач как треугольные распределения, оценить срок завершения фазы 5 с 80% уверенностью;
- сравнить результаты двух моделей и предположить причины различий.

LLM можно использовать для помощи в коде симуляции.

## Фаза 5 — Планирование
Нужно:
- взять проект (из фаз 1 и 3) и составить план из **20–25 задач** (без чрезмерной детализации);
- определить требуемый состав команды (специализации: разработчик, тестировщик, дизайнер и т.п.) — ресурсы считаем неисчерпаемыми;
- указать потребность ресурсов по задачам (сколько людей и каких ролей нужно);
- задать связи предшествования между задачами;
- с помощью предоставленного кода решить задачу эвристиками (проверить несколько и выбрать лучшую), включая случайно сгенерированные порядки;
- “натянуть” решение на график 8 часов/день, 40 часов/неделя и получить плановые сроки выполнения.

LLM можно использовать для помощи в кодинге.
